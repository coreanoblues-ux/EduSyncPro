역할

당신은 내 Replit 프로젝트의 리포지토리 전반을 먼저 스캔하여 현재 스택(프론트/백엔드/DB), 라우트, 상태관리, 스키마를 파악하고, 아래 요구사항을 충족하도록 구체적 코드 수정과 테스트를 직접 제안/반영한다. 임시 하드코딩, “나중에” 처리, 불명확한 가정 금지.

현재 문제 요약 (필수 해결)

새 학생 등록 시 에러 발생

재현: “새학생 등록” 플로우 마지막 단계에서 에러 토스트/다이얼로그가 떠서 등록 실패.

목표: 원인 추적(프론트 유효성/네트워크/백엔드 검증/DB 제약) → 안전하게 등록 성공, 실패 시 필드별 구체 메시지 표출.

대시보드 학생 카드의 수강료 금액이 잘못 표시

현상: 등록된 학생의 카드 금액이 실제 데이터 미반영, 임의로 150,000원(15만원)으로 고정되어 보임.

목표: 하드코딩 제거. 금액은 “해당 학생의 현재 활성 반/등록(Enrollment)과 연결된 실제 수강료/할인/미납액”으로 계산해 표기. 상태 변경(수납, 할인, 반 변경) 시 즉시 반영.

수납관리 섹션 UX 전면 개선(가장 중요)

기본 목록: 반 배정된 등록 학생 전체 표시.

정렬: 미납 학생이 최상단 → 그 다음 완납. 미납 내에서는 연체일수(또는 기한 임박) 우선, 그 다음 미납액 내림차순.

열 구성(예시): 이름 | 반명 | 청구액 | 납부액 | 미납액 | 마감일 | 상태(미납/부분납/완납) | 액션(수납/영수증/기록)

라벨링: 각 이름 옆에 반명 상시 노출(배지/텍스트).

검색/필터:

학생 이름 검색 입력(디바운스 300ms, 초성/부분일치 허용 가능하면 적용)

반 선택 필터(드롭다운/토글)

두 조건 동시 적용 가능.

상태 변경(수납 입력/취소/할인) 시 낙관적 업데이트 또는 재조회로 바로 반영.

수행 순서

리포 스캔: 패키지/의존성, 실행 스크립트, .env, API 라우트, DB 스키마(테이블/모델), 프론트 상태관리(React/Redux/Zustand 등), 금액 표기/계산 로직 위치 파악.

재현 테스트 자동화:

유닛/통합 테스트(백엔드 검증), E2E(프론트 → 백엔드 등록 → 대시보드 반영) 시나리오 추가.

최소: “정상 등록/필드 누락/중복/서버 오류” 케이스.

원인 분석 & 수정

등록 실패: 스키마 제약(UNIQUE/NOT NULL/외래키), DTO/스키마 유효성(Zod/Yup/Joi/Express-Validator 등), 트랜잭션 처리, 반환 페이로드(201/200와 본문) 맞춤.

금액 15만원 고정: 모든 하드코딩 상수 탐지(150000, 150,000, ₩150000) → 제거. 금액은 Enrollment → Pricing(수강료) → 할인/장학 → 납부내역(Payments) 를 이용해 계산.

수납관리: 정렬/검색/필터를 서버 쿼리(가능하면)로 처리하고, 프론트에서 안정적으로 표시. 페이지네이션 또는 무한스크롤 허용.

UI/UX 구현 지침

테이블 헤더 고정, 열 정렬 가능(상태/미납액).

상태 뱃지 색상 일관(미납=경고, 부분납=정보, 완납=성공).

액션은 명확한 호출 버튼과 확인 모달. 수납 폼에서는 금액, 결제수단, 비고, 자동 영수증 옵션.

반명은 이름 오른쪽 작은 배지로 상시 노출.

성능/정합성

금액/통화는 KRW로 천단위 구분 표기.

서버에서 합산 계산 시 정수(원 단위)로 처리해 부동소수 오차 방지.

인덱스(학생명, 반ID, 상태, 마감일) 추가로 조회 최적화.

로그 & 에러 처리

등록 실패 시 백엔드 구체 에러코드/메시지(예: DUPLICATE_STUDENT_PHONE, INVALID_CLASS_ID).

프론트는 필드별 인라인 에러와 상단 요약 토스트 모두 제공.

반영 후 회귀 테스트: 대시보드/수납관리/등록 플로우 전부 재검증.

데이터/계산 규칙(참고 모델; 실제 스키마에 맞춰 매핑)

students(id, name, phone, ... )

classes(id, name, ... )

enrollments(id, student_id, class_id, start_date, status)

pricing(id, class_id, base_fee) 또는 tuition(id, enrollment_id, list_price, discount_amount)

payments(id, enrollment_id, amount, paid_at, method)

미납액 = 청구액(리스트가격 - 할인) − 누적 납부액(sum(payments.amount))

대시보드 학생 카드 금액 = 해당 학생의 활성 enrollment(최근/현재 반) 기반 미납액(또는 청구액) 표기. 하드코딩 금지.

API/상태 계약(예시; 현재 코드에 맞춰 어댑트)

POST /api/students: 201 + { student, enrollment? } 반환. 유효성 실패 시 400 + 필드별 코드.

GET /api/dashboard/students?includeBalances=true: 각 학생별 최신 활성 enrollment와 금액 필드 포함.

GET /api/payments?assigned=true&status=unpaid|partial|paid&classId=...&q=...: 수납관리용 목록. 서버 정렬: status, overdueDays DESC, outstanding DESC.

POST /api/payments: 수납 등록 후 최신 잔액 반환.

수락(완료) 기준 (Acceptance Criteria)

A. 새학생 등록

 모든 필수 필드 입력 시 즉시 성공, 화면에 성공 토스트 + 목록/대시보드 자동 갱신.

 필수 누락/중복/잘못된 반ID 등은 필드 단위 에러 메시지와 포커스 이동.

 서버/클라이언트 로그에 스택/에러코드 남김.

B. 대시보드 금액

 학생 카드 금액이 실제 DB 계산값과 일치.

 150,000원 하드코딩 흔적 전부 제거(검색 기준 포함).

 수납/할인/반 변경 후 즉시 갱신.

C. 수납관리 UX

 기본 목록이 반 배정된 학생 전체를 포함하고, 미납 → 부분납 → 완납 순으로 보임.

 각 이름 옆에 반명 표시.

 이름 검색 + 반 필터가 동시에 동작, 300ms 디바운스, 한글 부분/초성 일치 지원 가능 시 적용.

 액션(수납/영수증/기록)이 분명하고, 수납 등록 시 잔액/상태가 즉시 업데이트.

D. 테스트

 유닛/통합/E2E 테스트 스위트가 통과하며, 재현 케이스가 모두 녹아 있음.

 주요 계산(미납액, 정렬, 필터)이 테스트로 보장.

추가 제약

하드코딩 금지(150000, 150,000, ₩150,000 등 문자열/숫자 전수 검사).

통화/날짜 포맷은 KRW/한국 로케일 기준.

가능하면 서버 정렬/필터 우선, 프론트는 보조.

변경으로 인한 성능 저하 금지(N+1 쿼리 방지, 필요한 인덱스 부여).

먼저 수행할 점검 체크리스트

 패키지 매니저/빌드 스크립트 확인(npm/pnpm/yarn 혹은 pip/poetry).

 .env 키(DB_URL 등) 로딩 및 마이그레이션 상태.

 등록 폼의 클라이언트 검증 규칙과 서버 검증 규칙 정합성.

 대시보드 금액 계산 경로(서버 계산 vs 프론트 계산) 단일화.

 수납 트랜잭션 원자성(실패 시 롤백).
